# PRD: 예상 면접질문

- 문서 버전: v1.0
- 작성일: 2026-07-14
- 대상 화면: 채용 공고 상세 화면 내 "예상 면접질문" 섹션 (+ 연습 모드)
- 연관 문서: [[PRD.md]] (메인 페이지 PRD), [[PRD-job-detail.md]] (채용 공고 상세 PRD)

## 0. 배경 및 전제

- 현재 채용 공고 데이터는 원티드(Wanted) OpenAPI에서 수집한다 (`.env`의 `client_id`/`client_secret` → `scripts/sync-wanted-dev-jobs.mjs` → `data/wanted-jobs.json` → Supabase `jobs` 테이블).
- `supabase/functions/sync-jobs/index.ts` 주석에 명시된 대로, 원티드 실제 API 응답에는 **급여·기술스택·학력·면접질문 정보가 없다**. 확보 가능한 직무 분류 필드는 `category_parent`(대분류, 예: "개발"), `category_children`(소분류, 예: "서버", "프론트엔드", "iOS")뿐이다.
- 따라서 공고 원문에서 면접질문을 직접 추출/생성할 수 없고, 본 기능은 **직무 소분류(`category_children`) 단위로 매핑된 질문 뱅크**를 기반으로 한다.
- `.env`에는 현재 원티드 API 인증 정보만 존재하며 LLM(예: Claude) API 키는 없다. 향후 공고별 맞춤 질문을 AI로 생성하려면 별도 API 키 추가가 선행되어야 하므로, 초기 버전은 **정적 큐레이션 질문뱅크**로 범위를 한정한다(3장 참고).

## 1. 핵심 기능 목록 (Feature List & Data Fields)

### 1.1 질문 뱅크 데이터

| 필드명 | 타입 | 설명 | 필수 |
|---|---|---|---|
| `question_id` | string | 질문 고유 ID | Y |
| `category_children` | string | 매핑 대상 직무 소분류 (기존 `job.category_children`과 매칭, 예: "서버", "프론트엔드") | Y |
| `question_type` | enum [기술, 인성, 경험] | 질문 유형 | Y |
| `question_text` | string | 질문 본문 | Y |
| `answer_tip` | string | 답변 방향/체크포인트 가이드 (모범답안 아님) | Y |
| `difficulty` | enum [기본, 심화] | 난이도 | N |
| `source_note` | string | 출처/근거 메모 (예: "실무 공통 질문", 사내 참고용, 화면 비노출) | N |

**동작 요구사항**: 하나의 `category_children`에 여러 질문이 매핑될 수 있으며(1:N), 질문 유형별로 최소 3개 이상 확보를 목표로 한다. 매핑되는 질문이 0개인 소분류는 섹션 자체를 숨기고 "공통 질문"(직무 무관 인성 질문 세트)으로 대체 노출한다.

### 1.2 예상 면접질문 섹션 (공고 상세 내)

| 필드명 | 타입 | 설명 | 필수 |
|---|---|---|---|
| `matched_category` | string | 현재 공고의 `category_children` 중 질문이 매핑된 항목 | Y |
| `question_list` | array\<question\> | 매칭된 질문 목록 (기술/인성/경험 그룹핑) | Y |
| `question_count_by_type` | object | 유형별 개수 (탭/카운트 배지 노출용) | N |
| `is_fallback_common` | boolean | 직무 매핑 질문이 없어 공통 질문으로 대체됐는지 여부 | N |

**동작 요구사항**: 유형별 탭(전체/기술/인성/경험)으로 필터링. 기본 노출은 접힌 리스트(질문만 보임), 클릭 시 `answer_tip` 아코디언 확장. 공고 상세 모달의 "채용 전형과정" 섹션 아래에 위치.

### 1.3 연습 모드 (카드 뒤집기)

| 필드명 | 타입 | 설명 | 필수 |
|---|---|---|---|
| `practice_deck` | array\<question\> | 연습 대상 질문 목록 (현재 매칭된 질문 전체 또는 스크랩한 질문만) | Y |
| `current_index` | number | 현재 카드 인덱스 | Y |
| `is_revealed` | boolean | 답변 팁 공개 여부 (카드 클릭/스페이스로 토글) | Y |
| `user_note` | string | 사용자가 직접 작성하는 답변 메모 (로컬 저장) | N |

**동작 요구사항**: "연습해보기" 버튼으로 진입, 질문만 먼저 보여주고 클릭 시 `answer_tip` 공개. 이전/다음 카드 이동. 로그인 상태에서는 `user_note`를 이력서 탭과 동일한 저장 방식(Supabase)으로 보존, 비로그인 시 미제공.

### 1.4 질문 스크랩

| 필드명 | 타입 | 설명 | 필수 |
|---|---|---|---|
| `is_bookmarked` | boolean | 질문 스크랩 여부 | N |
| `bookmarked_at` | datetime | 스크랩 시각 | N |

**동작 요구사항**: 로그인 사용자만 이용 가능(기존 이력서 탭과 동일한 인증 요구 패턴 재사용). 스크랩한 질문은 이력서 탭 또는 별도 "내 질문노트"에서 모아보기 가능(범위는 P2에서 확정).

## 2. 사용자 시나리오

### 시나리오 1 — 지원 전 예상 질문 확인
- **사용자**: 관심 공고에 지원을 고민 중인 구직자
- **목표**: 해당 직무에서 자주 나오는 질문 유형을 미리 파악
- **흐름**: 공고 상세 진입 → 전형과정 확인 → "예상 면접질문" 섹션 스크롤 → 기술/인성/경험 탭 전환하며 질문 목록 확인 → 관심 질문 클릭해 답변 팁 확인
- **기대 결과**: 외부 검색 없이 공고 화면 안에서 직무 맞춤 질문 파악

### 시나리오 2 — 연습 모드로 실전 대비
- **사용자**: 면접 일정이 잡힌 지원자
- **목표**: 질문을 보고 답을 먼저 떠올린 뒤 팁과 비교
- **흐름**: "연습해보기" 버튼 클릭 → 카드에 질문만 노출 → 스스로 답변 구상 → 카드 클릭해 `answer_tip` 공개 → 다음 카드로 이동, 필요 시 답변 메모 작성
- **기대 결과**: 정답 암기가 아닌 자가 점검 방식으로 준비, 로그인 시 메모가 남아 재방문 시에도 이어서 확인 가능

### 시나리오 3 — 직무 매핑 질문이 없는 경우
- **사용자**: 질문 뱅크에 아직 등록되지 않은 소수 직무(`category_children`) 공고를 보는 구직자
- **목표**: 빈 화면이 아닌 최소한의 유용한 정보를 받고 싶음
- **흐름**: 상세 화면 진입 → 직무 매핑 질문 없음 → "공통 질문(인성/경험)" 자동 대체 노출 + "이 직무의 질문이 아직 준비되지 않았어요" 안내 문구
- **기대 결과**: 빈 섹션 노출을 방지하고 최소 콘텐츠 보장

### 시나리오 4 — 스크랩한 질문 모아보기
- **사용자**: 여러 공고를 비교하며 준비하는 구직자
- **목표**: 여러 공고에서 스크랩한 질문을 한곳에서 복습
- **흐름**: 여러 공고 상세를 순회하며 질문 스크랩 → 이력서 탭(또는 별도 진입점)에서 스크랩 목록 확인 → 연습 모드로 진입해 복습
- **기대 결과**: 공고 단위로 흩어진 질문을 하나의 학습 흐름으로 재사용

## 3. 데이터 모델 변경 (전제 작업)

| 대상 | 변경 내용 |
|---|---|
| Supabase 신규 테이블 `interview_questions` | `question_id`, `category_children`, `question_type`, `question_text`, `answer_tip`, `difficulty`, `source_note` 컬럼 구성. 초기 데이터는 직무 소분류별 수동 큐레이션(운영자 입력)으로 시딩 |
| Supabase 신규 테이블 `interview_bookmarks` | `user_id`, `question_id`, `bookmarked_at`, `user_note` — 로그인 사용자별 스크랩/메모 저장 (기존 이력서 관련 테이블과 동일한 RLS 정책 적용) |
| `data.js` | 로컬 프로토타입 단계에서는 `EVENTS`/`COMMUNITY_POSTS`와 동일한 방식의 `INTERVIEW_QUESTIONS` 목업 배열 추가 (소분류 → 질문 배열 매핑) |
| `app.js` | 공고 상세 모달에 "예상 면접질문" 섹션 렌더링 함수 추가. `job.category_children`으로 `interview_questions` 조회/매칭하는 로직 필요 |
| `.env` | **P0~P2 범위에서는 변경 불필요** (정적 질문뱅크만 사용). P3(AI 맞춤 질문 생성)로 확장 시 LLM API 키(예: `ANTHROPIC_API_KEY`) 신규 추가 필요 — 현재 파일에는 원티드 인증 정보(`client_id`, `client_secret`)만 존재함을 확인함 |

**미해결 이슈**: 원티드 API의 `category_children`은 다건(배열)일 수 있어, 질문 매칭 시 우선순위(어떤 소분류를 대표로 쓸지) 기준이 필요함 — 별도 확인 필요.

## 4. 개발 우선순위

| 우선순위 | 범위 | 포함 기능 | 근거 |
|---|---|---|---|
| **P0 (MVP)** | 질문 노출 골격 | `interview_questions` 정적 질문뱅크(직무 소분류 매핑), 공고 상세 내 섹션 노출, 유형별 탭(전체/기술/인성/경험) | 원티드 API에 면접질문 데이터가 없으므로 큐레이션 데이터만으로 구현 가능한 최소 골격. 신규 API 연동 없이 착수 가능 |
| **P1** | 학습 보조 기능 | 연습 모드(카드 뒤집기, 이전/다음 이동), 매핑 없음 시 공통 질문 대체 로직 | 질문뱅크 데이터가 갖춰진 이후 UX 개선 성격으로 후순위 배치 |
| **P2** | 개인화 저장 | 질문 스크랩, 로그인 사용자 답변 메모 저장(`interview_bookmarks`), 스크랩 모아보기 | 로그인/Supabase 연동이 필요해 인증 플로우(이력서 탭 인증 재사용) 안정화 이후 진행 |
| **P3 (후속)** | AI 맞춤 질문 생성 | 공고 설명 기반 LLM 질문 생성, 사용자 답변에 대한 AI 피드백 | `.env`에 LLM API 키가 없어 신규 키 발급/연동이 선행되어야 하며, 프롬프트 설계·비용 정책 등 추가 검토 필요 |

**우선순위 판단 기준**: (1) 원티드 API의 데이터 한계(면접질문 필드 부재)를 인정하고 정적 큐레이션으로 먼저 가치를 제공, (2) 로그인·저장이 필요한 기능은 기존 이력서 탭의 인증/저장 패턴이 검증된 이후 확장, (3) AI 생성 기능은 별도 API 키 확보와 비용/품질 검증이 선행되어야 하므로 명확히 후순위로 분리.
