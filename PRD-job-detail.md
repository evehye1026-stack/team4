# PRD: 채용 공고 상세 화면

- 문서 버전: v1.0
- 작성일: 2026-07-14
- 대상 화면: 채용 공고 카드 클릭 시 노출되는 상세 화면
- 연관 문서: [[PRD.md]] (메인 페이지 PRD)

---

## 0. 배경 및 전제

현재 `app.js`의 `renderJobCard()`는 카드 클릭 시 `window.open(job.url, "_blank")`로 원티드 등 **외부 사이트**로 이동시킨다(app.js:73). 본 PRD는 이 클릭 동작을 **서비스 내부 상세 화면**으로 대체하는 것을 전제로 한다.

또한 `jobs.csv` / `data.js`의 현재 데이터 모델에는 아래 항목이 없어 **신규 필드 추가(데이터 모델 확장)** 가 선행되어야 한다:

- 합격보상, 복지 및 혜택, 채용 전형과정, 고용형태, 포지션 상세(주요업무/자격요건/우대사항 세분화), 마감일, 근무지 상세좌표

기존에 있는 `city`, `district`, `subcategory`, `skills`, `years`, `education`, `position`, `workDays` 등은 그대로 재사용한다.

**노출 방식(모달 vs 페이지)**: 현재 앱은 라우팅 없는 단일 페이지(SPA) 구조이며, 검색 드로어(`search-drawer`)와 동일하게 오버레이 패턴을 이미 사용 중이다. 별도 URL 라우팅 도입 비용을 줄이기 위해 **모달(오버레이) 방식**을 기본안으로 제안한다. (근거는 3장 참고)

## 1. 핵심 기능 목록 (Feature List & Data Fields)

### 1.1 헤더 영역

| 필드명 | 타입 | 설명 | 필수 |
|---|---|---|---|
| `job_id` | string | 공고 고유 ID (기존 `id`) | Y |
| `company_name` | string | 회사이름 (기존 `company`) | Y |
| `company_logo_url` | string | 회사 로고 (기존 `logo`, 현재 이모지 → 추후 이미지 URL 전환 여지) | N |
| `job_title` | string | 포지션명 (기존 `name`) | Y |
| `job_category` | string | 구직 분야 (기존 `subcategory`: 서버/프론트엔드/iOS 등) | Y |
| `employment_type` | enum [정규직, 계약직, 인턴, 프리랜서] | 고용형태 — **신규 필드** | Y |
| `deadline_date` | date \| null | 마감일 — **신규 필드**. null이면 상시채용 | Y |
| `is_always_open` | boolean | 상시채용 여부 (deadline_date가 null일 때 true) | Y |
| `dday_label` | string (계산값) | "D-7" / "상시채용" / "마감" 자동 계산 표시 | Y |

**동작 요구사항**: 마감일이 3일 이내면 배지 강조(경고색). `deadline_date`가 지난 경우 "마감된 공고" 상태 배너 노출, 지원 관련 CTA 비활성화.

---

### 1.2 합격보상

| 필드명 | 타입 | 설명 | 필수 |
|---|---|---|---|
| `has_hiring_reward` | boolean | 합격보상 제공 여부 — **신규 필드** | Y |
| `hiring_reward_amount` | number | 보상 금액(원) | N (has_hiring_reward=true일 때 Y) |
| `hiring_reward_condition` | string | 지급 조건 (예: "입사 후 3개월 근속 시 지급") | N |

**동작 요구사항**: `has_hiring_reward`가 false면 섹션 자체를 숨김(빈 섹션 노출 금지). 금액은 천 단위 콤마 포맷.

---

### 1.3 복지 및 혜택

| 필드명 | 타입 | 설명 | 필수 |
|---|---|---|---|
| `benefits` | array\<{icon, title, description}\> | 복지 항목 리스트 — **신규 필드** | Y |
| `benefits[].icon` | string | 아이콘(이모지 또는 아이콘 키) | N |
| `benefits[].title` | string | 항목명 (예: "재택근무", "간식 제공") | Y |
| `benefits[].description` | string | 부가 설명 | N |

**동작 요구사항**: 그리드형(2~3열) 카드 나열. 항목 0개 시 "등록된 복지 정보가 없습니다" 안내 문구.

---

### 1.4 채용 전형과정

| 필드명 | 타입 | 설명 | 필수 |
|---|---|---|---|
| `hiring_process_steps` | array\<{step_order, step_name, step_description}\> | 전형 단계 목록 — **신규 필드** | Y |
| `hiring_process_steps[].step_order` | number | 단계 순서 (1, 2, 3…) | Y |
| `hiring_process_steps[].step_name` | string | 단계명 (예: 서류전형, 1차 면접, 처우협의) | Y |
| `hiring_process_steps[].step_description` | string | 단계별 상세 설명 | N |

**동작 요구사항**: 가로 스텝퍼(단계 순서대로 화살표 연결) UI. 모바일에서는 세로 스텝퍼로 반응형 전환.

---

### 1.5 포지션 상세

| 필드명 | 타입 | 설명 | 필수 |
|---|---|---|---|
| `main_tasks` | string[] | 주요업무 목록 — **신규 필드** | Y |
| `qualifications` | string[] | 자격요건 목록 — **신규 필드** | Y |
| `preferred_qualifications` | string[] | 우대사항 목록 — **신규 필드** | N |
| `tech_stack` | string[] | 기술스택 (기존 `skills` 재사용) | N |

**동작 요구사항**: 각 항목은 불릿 리스트. `preferred_qualifications`가 없으면 섹션 숨김.

---

### 1.6 근무지역

| 필드명 | 타입 | 설명 | 필수 |
|---|---|---|---|
| `work_location_l1` | string | 근무지 시/도 (기존 `city`) | Y |
| `work_location_l2` | string | 근무지 시/군/구 (기존 `district`) | Y |
| `work_address_detail` | string | 상세 주소 (건물명 등) — **신규 필드** | N |
| `map_lat` / `map_lng` | number | 지도 좌표 — **신규 필드** (기존 `CITY_COORDS`는 시 단위 중심좌표라 정밀도 부족, 회사별 좌표 추가 필요) | Y |

**동작 요구사항 — "지도로 위치보기" 버튼**:
1. 기본 상태에서는 주소 텍스트만 노출, 지도는 렌더링하지 않음(성능 고려).
2. 버튼 클릭 시 해당 섹션 하단에 Leaflet 지도 인라인 토글 오픈(아코디언 방식), 재클릭 시 닫힘.
3. 지도 오픈 시 `map_lat/map_lng` 좌표에 마커 1개 표시, 기본 줌 레벨 15~16.
4. 기존 지역별 탭의 지도(`initMap()`)와 동일한 Leaflet 인스턴스/타일 서버(OpenStreetMap) 재사용.
5. 모달 내부에서 지도를 늦게(지연) 초기화해야 하는 이슈(현재 `mapInitialized` 패턴과 동일)에 유의 — 토글 오픈 시점에 `map.invalidateSize()` 호출 필요.

---

### 1.7 화면 열기/닫기 인터랙션

| 항목 | 스펙 |
|---|---|
| 트리거 | 공고 카드 클릭 (기존 `window.open(job.url)` 대체) |
| 노출 방식 | 모달 오버레이 (배경 딤 처리 + 중앙/우측 슬라이드 패널) |
| 닫기 | 우측 상단 ✕ 버튼, 배경 클릭, `Esc` 키 — 기존 `search-drawer` 패턴과 동일 |
| 접근성 | 열림 시 포커스를 모달 내부로 이동, 닫힘 시 트리거였던 카드로 포커스 복귀, `aria-hidden`/`aria-modal` 속성 적용 |
| URL 반영 | (선택) 새로고침/공유 대응을 위해 `?job=<job_id>` 쿼리스트링 반영 권장 — 초기 버전은 생략 가능 |

## 2. 사용자 시나리오

### 시나리오 1 — 상세 조건 확인 후 지원 여부 결정
- **사용자**: 리스트에서 관심 공고를 발견한 구직자
- **목표**: 지원 전 합격보상·복지·전형과정을 한 번에 확인
- **흐름**: 카드 클릭 → 상세 모달 오픈 → 헤더(회사명/구직분야/고용형태/마감일) 확인 → 스크롤하며 합격보상 → 복지 및 혜택 → 전형과정 → 포지션 상세 순으로 확인
- **기대 결과**: 외부 페이지 이동 없이 서비스 내에서 의사결정에 필요한 정보를 모두 확인

### 시나리오 2 — 근무지 위치 확인
- **사용자**: 통근 거리를 중요하게 보는 구직자
- **목표**: 실제 근무지가 집/역에서 가까운지 확인
- **흐름**: 상세 화면 하단 근무지역 섹션 → "지도로 위치보기" 버튼 클릭 → 인라인 지도 오픈, 정확한 위치 마커 확인 → 필요 시 재클릭으로 지도 닫고 다른 정보 계속 탐색
- **기대 결과**: 페이지 이탈 없이 위치 확인, 지도가 다른 섹션 스크롤을 방해하지 않음(토글형이라 기본 숨김)

### 시나리오 3 — 마감 임박/마감된 공고 확인
- **사용자**: 여러 공고를 비교 중인 구직자
- **목표**: 지원 가능 여부(마감 전인지) 빠르게 인지
- **흐름**: 상세 화면 진입 → 헤더의 D-day 배지 확인 (예: "D-3" 경고색 강조, 또는 "마감된 공고" 배너)
- **기대 결과**: 마감 임박/마감 여부를 텍스트를 정독하지 않아도 색상/배지로 즉시 인지

### 시나리오 4 — 모바일 환경에서 전형과정 확인
- **사용자**: 모바일로 접속한 구직자
- **목표**: 좁은 화면에서도 전형 단계를 순서대로 파악
- **흐름**: 모달이 풀스크린으로 전환 → 채용 전형과정 섹션에서 가로 스텝퍼 대신 세로형 타임라인으로 표시 → 단계별 설명 확인
- **기대 결과**: 데스크톱과 동일한 정보를 레이아웃만 다르게 제공받아 가독성 유지

## 3. 데이터 모델 변경 (전제 작업)

| 대상 | 변경 내용 |
|---|---|
| `jobs.csv` | 컬럼 추가: `employmentType`, `deadlineDate`, `hasHiringReward`, `hiringRewardAmount`, `hiringRewardCondition`, `mapLat`, `mapLng`, `addressDetail` |
| `data.js` / `csvToJobs()` | 위 신규 컬럼 파싱 로직 추가, 기본값 처리(예: `hasHiringReward` 없으면 false) |
| 신규 데이터 구조 | `benefits`, `hiringProcessSteps`, `mainTasks`, `qualifications`, `preferredQualifications`는 콤마/파이프 구분자로는 표현이 복잡한 배열-of-객체 구조라 **CSV 대신 별도 JSON(`job-details.json`) 또는 CSV 내 JSON 문자열 컬럼**으로 분리 관리 권장 |
| `app.js` | `renderJobCard()`의 클릭 핸들러를 `window.open(job.url)` → `openJobDetail(job.id)` (모달 오픈 함수)로 교체 |

**미해결 이슈**: 기존 카드에 있던 외부 지원 링크(`url`, 원티드 등)를 상세 화면 내 "지원하기" 버튼으로 유지할지 여부는 본 PRD 범위 밖이며 별도 확인 필요.

## 4. 개발 우선순위

| 우선순위 | 범위 | 포함 기능 | 근거 |
|---|---|---|---|
| **P0 (MVP)** | 핵심 정보 노출 골격 | 모달 오픈/닫기, 헤더(회사명/구직분야/고용형태/마감일), 포지션 상세(주요업무/자격요건), 근무지역 텍스트 표시 | 상세 화면의 기본 존재 목적(핵심 정보 열람)을 충족하는 최소 골격. 신규 텍스트/배열 필드만으로 구현 가능 |
| **P1** | 의사결정 보조 정보 | 합격보상 섹션, 복지 및 혜택, 채용 전형과정(스텝퍼) | 지원 결정에 영향을 주는 정보이나 신규 데이터 스키마(배열-of-객체) 설계 및 목업 데이터 입력이 선행되어야 함 |
| **P2** | 위치 시각화 | "지도로 위치보기" 버튼 + Leaflet 인라인 지도, 회사별 정밀 좌표(`mapLat/mapLng`) 데이터 확보 | 기존 지역 탭 지도와 별개로 회사 단위 좌표 데이터가 추가로 필요하고, 지도 지연 초기화 등 기술적 예외처리가 필요해 후순위 |
| **P3 (후속)** | 편의/공유 기능 | 마감 임박 강조 배지, URL 쿼리 반영(공유 가능한 링크), 지원하기 CTA 연동 | 초기 출시 이후 사용성 데이터 기반으로 추가 검토 |

**우선순위 판단 기준**: (1) 클릭 시 "정보가 보여야 한다"는 화면의 존재 목적을 먼저 충족, (2) 신규 데이터 스키마(배열/객체) 설계가 필요한 기능은 스키마 확정 이후로 후순위 배치, (3) 지도 등 외부 라이브러리 연동 및 좌표 데이터 확보가 필요한 기능은 별도 데이터 작업과 함께 진행.
